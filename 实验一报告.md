# 实验一：1.十六进制字符串转换为 Base64

## 题目描述

本题要求把给定的十六进制字符串转换成 Base64 表示。我的理解是，十六进制和 Base64 都不是加密算法本身，而是把同一段二进制数据用不同字符集显示出来的编码方式。题目给出的输入看起来是一长串十六进制字符，实际含义是一段原始字节；程序要先把每两个十六进制字符还原成一个字节，再按 Base64 的规则重新编码成可打印字符串。这里最重要的一点是不能直接把十六进制字符本身拿去做 Base64，而是要先回到原始 bytes。

## 过程

实验的背景来自 Cryptopals Set 1 Challenge 1，后续很多题都会反复使用“字节数组”这个中间层。十六进制编码中，1 个字符表示 4 bit，所以 2 个十六进制字符正好表示 1 byte。例如 `49` 表示一个字节，它的十进制值是 73，对应 ASCII 字符 `I`。Base64 的思路不同，它每次取 3 个字节，也就是 24 bit，然后把这 24 bit 分成 4 组，每组 6 bit。因为 6 bit 可以表示 0 到 63，所以刚好可以映射到 64 个 Base64 字符：`A-Z`、`a-z`、`0-9`、`+`、`/`。

实现时我先写了 `hex_to_bytes`，负责清理输入中的空白字符，并检查十六进制字符串长度是否为偶数，然后用 `bytes.fromhex` 得到原始字节。随后 `bytes_to_base64` 每次取 3 个字节，通过 `int.from_bytes(block, "big")` 把这几个字节看成一个大端整数。如果最后剩下 1 个或 2 个字节，就在低位补 0，让它仍然能凑成 24 bit，再根据缺少的字节数把结果末尾替换成 `=`。主要转换逻辑如下：

```python
for index in range(0, len(data), 3):
    block = data[index:index + 3]
    padding = 3 - len(block)
    twenty_four_bits = int.from_bytes(block, "big") << (padding * 8)

    for shift in (18, 12, 6, 0):
        six_bits = (twenty_four_bits >> shift) & 0b111111
        result.append(BASE64_ALPHABET[six_bits])

    if padding:
        result[-padding:] = "=" * padding
```

这段代码的关键是位运算。`>>` 用来把目标的 6 bit 移到最低位，`& 0b111111` 用来只保留最低 6 bit。这样每组 6 bit 都能变成 Base64 字符表里的一个下标。最后运行程序，输出结果为 `SSdtIGtpbGxpbmcgeW91ciBicmFpbiBsaWtlIGEgcG9pc29ub3VzIG11c2hyb29t`，与题目要求一致。

## 总结

通过这个实验我更清楚地区分了“原始字节”和“编码字符串”。十六进制、Base64 都只是展示同一批 bytes 的方式，本身不提供保密性。写代码时最容易犯的错误是把字符串当成数据本身处理，比如直接对 `"4927..."` 这串字符做 Base64，这样得到的会是十六进制文本的编码，而不是原始消息的编码。解决方法是始终先转换成 bytes，在 bytes 层完成真正的数据处理，最后只在输出时转换成十六进制或 Base64 这样的可读形式。

# 实验一：2.Fixed XOR

## 题目描述

本题要求编写一个函数，接收两段长度相同的缓冲区数据，并逐字节计算它们的异或结果。题目中给出的两段数据使用十六进制字符串表示，因此程序不能直接对字符串字符做运算，而是要先把十六进制还原成真实的字节序列。还原之后，两段字节在相同下标位置上一一进行 XOR 运算，最后再把得到的结果转回十六进制字符串。我的理解是，这道题主要训练的是在密码学中把“显示用的编码”和“真正参与运算的字节”区分开，同时熟悉异或这种最常见的底层运算。

## 过程

异或运算记作 XOR，在 Python 中使用 `^` 表示。它的基本规则是：相同为 0，不同为 1，也就是 `0 ^ 0 = 0`、`1 ^ 1 = 0`、`0 ^ 1 = 1`、`1 ^ 0 = 1`。扩展到一个字节时，计算机会对这个字节中的 8 个 bit 分别执行同样的规则。例如一个字节 `0x1c` 和另一个字节 `0x68` 做异或，实质上是 `00011100 ^ 01101000 = 01110100`，结果就是 `0x74`。题目中的两段十六进制数据长度相同，所以可以把它们解码为两个等长的 bytes，然后按位置配对计算。

我实现时把程序分成三层：`hex_to_bytes` 负责去掉输入里的空白并完成十六进制解码；`fixed_xor` 只处理 bytes，并检查两段数据长度是否相等；`fixed_xor_hex` 把前两步串起来，方便直接输入题目中的十六进制字符串。核心代码如下：

```python
def fixed_xor(left: bytes, right: bytes) -> bytes:
    if len(left) != len(right):
        raise ValueError("buffers must have the same length")

    return bytes(left_byte ^ right_byte for left_byte, right_byte in zip(left, right))
```

这里使用 `zip(left, right)` 同时遍历两段 bytes 的每一个位置，`left_byte ^ right_byte` 得到该位置的异或结果，外层 `bytes(...)` 再把所有结果组装回新的字节序列。最后运行程序，输入 `1c0111001f010100061a024b53535009181c` 和 `686974207468652062756c6c277320657965`，得到输出 `746865206b696420646f6e277420706c6179`，与题目给出的结果一致。

## 总结

这次实验让我更直观地理解了 XOR 在密码学中的作用。XOR 的特点是可逆：如果 `C = A ^ B`，那么 `C ^ B` 又会回到 `A`，因为同一个值异或两次会互相抵消。这个性质使它经常出现在流密码、一次一密、分组密码内部轮函数等场景中。不过 XOR 本身不是安全的加密算法，它只是一个基础运算，安全性取决于密钥是否随机、是否只使用一次以及整体算法设计是否可靠。写代码时我碰到的关键问题是不要把十六进制文本本身当作明文数据处理，正确做法是先转成 bytes，在 bytes 层进行异或，再把结果编码成十六进制方便查看。

# 实验一：3.Single-byte XOR cipher

## 题目描述

这道题给出了一段十六进制编码后的密文，并说明它是由某一段英文明文和一个固定的单字符密钥逐字节异或得到的。我的理解是，题目并不是要求直接“解码”十六进制，因为十六进制只负责把字节显示成可读字符；真正需要破解的是异或加密。单字节 XOR 的特点是整段消息都反复使用同一个 1 byte 密钥，所以密钥空间只有 256 种。只要把这 256 种可能全部试一遍，再判断哪一个结果最像正常英文，就可以找出密钥和明文。

## 过程

XOR 运算有一个很重要的性质：如果 `C = P xor K`，那么 `P = C xor K`。原因是同一个数异或两次会抵消，即 `K xor K = 0`，而任何数和 0 异或仍然等于它本身。所以加密时明文字节和密钥字节异或，解密时密文字节再和同一个密钥字节异或，就能恢复原来的明文。题目中的密钥只有一个字节，因此程序不用猜很大的密钥空间，只需要从 `0x00` 到 `0xff` 依次尝试。

实现时我先用 `bytes.fromhex` 把题目给出的十六进制字符串还原成密文字节。随后 `xor_with_single_byte` 对每个密文字节执行 `byte ^ key`，得到某个候选密钥下的明文。关键问题在于如何判断候选明文“像不像英文”。我使用了英文字符频率作为评分依据：空格、`e`、`t`、`a`、`o`、`i`、`n` 等常见字符出现时加分，普通可打印符号少量扣分，不可打印字符扣较多分。这样做的依据是自然语言并不是随机字节流，英文文本中空格和高频字母的比例通常比较稳定，而错误密钥解出的结果往往会出现乱码或很奇怪的字符分布。

主要实现代码如下：

```python
def break_single_byte_xor(ciphertext: bytes) -> tuple[int, bytes, float]:
    best_key = 0
    best_plaintext = b""
    best_score = float("-inf")

    for key in range(256):
        plaintext = xor_with_single_byte(ciphertext, key)
        score = score_english(plaintext)
        if score > best_score:
            best_key = key
            best_plaintext = plaintext
            best_score = score

    return best_key, best_plaintext, best_score
```

运行 `single_byte_xor.py` 后，程序输出的密钥字节是 `88`，对应 ASCII 字符 `'X'`，解出的明文是 `Cooking MC's like a pound of bacon`。这个结果能被评分函数选中，是因为它含有正常英文单词、空格和常见字母，而其他密钥得到的候选文本要么没有语义，要么包含大量不常见字符。

## 总结

这道题让我体会到，密码算法的安全性不能只看“用了异或”这种操作本身，还要看密钥空间和密钥使用方式。XOR 是很多密码算法里的基础运算，但单字节密钥太短，只有 256 种可能，暴力穷举几乎没有成本。破解时用到的技巧是把密码学问题转化成搜索问题：先枚举所有可能密钥，再用语言统计特征筛选结果。写代码时我一开始容易把十六进制字符串和真正的密文字节混在一起理解，后来明确了流程应该是“十六进制文本 -> 密文字节 -> XOR 尝试 -> 明文评分”，思路就清楚很多。

# 实验一：4.Detect single-character XOR

## 题目描述

本题给出了一组长度相同的十六进制字符串，其中只有一行是真正由“单字符 XOR”加密得到的密文，其余行可以看作干扰项。我的理解是，这道题比第 3 题多了一层筛选：第 3 题只需要对一段密文尝试 256 个可能密钥，而本题要对文件中的每一行都做同样的尝试，再从所有候选结果中找出最像英文明文的一行。因此程序的目标不是单纯解出某一段固定密文，而是同时判断“哪一行是密文”和“它的密钥是什么”。

## 过程

这道题仍然建立在 XOR 的可逆性上。若加密过程是 `C = P xor K`，其中 `K` 是一个固定的 1 byte 密钥，那么解密时只要再次异或同一个 `K`，就有 `P = C xor K`。由于单字节密钥最多只有 256 种取值，可以直接穷举，不需要复杂的数学推导。第 4 题的关键变化在于候选密文变多了，所以整体搜索空间变成“文件行数 × 256”。本实验数据共有 327 行，每行尝试 256 个密钥，总共也只有几万次异或运算，对计算机来说非常小。

实现时我复用了第 3 题中的 `hex_to_bytes`、`break_single_byte_xor` 和英文评分函数。程序先逐行读取 `detect_single_character_xor.txt`，把每一行从十六进制文本转换成真实密文字节；随后对该行调用 `break_single_byte_xor`，得到这一行在最佳密钥下的候选明文和分数。最后只保留全局分数最高的候选项。核心代码如下：

```python
def detect_single_character_xor(lines: list[str]) -> Candidate:
    best_candidate: Candidate | None = None

    for line_number, line in enumerate(lines, start=1):
        ciphertext_hex = line.strip()
        if not ciphertext_hex:
            continue

        ciphertext = hex_to_bytes(ciphertext_hex)
        key, plaintext, score = break_single_byte_xor(ciphertext)
        candidate = Candidate(line_number, ciphertext_hex, key, plaintext, score)

        if best_candidate is None or candidate.score > best_candidate.score:
            best_candidate = candidate

    if best_candidate is None:
        raise ValueError("no ciphertext lines were found")

    return best_candidate
```

英文评分的思路是利用自然语言的统计特征。正常英文中空格、`e`、`t`、`a`、`o`、`i` 等字符出现得比较多，而随机字节或错误密钥解出的文本往往包含大量不可打印字符，或者字符分布很奇怪。程序不需要真正“理解”英文，只要给常见英文字符加分、给不可打印字符扣分，就能让正确明文在大量候选中排到最前面。运行 `detect_single_character_xor.py` 后，得到的结果是第 171 行：

```text
line number: 171
ciphertext: 7b5a4215415d544115415d5015455447414c155c46155f4058455c5b523f
key byte: 53 (0x35, '5')
plaintext: Now that the party is jumping
score: 2.0532
```

可以看到，这一行使用的单字节密钥是 `0x35`，对应 ASCII 字符 `'5'`。密文中的每个字节都与 `0x35` 再异或一次后，恢复出了明文 `Now that the party is jumping`。这个结果也说明，单字节 XOR 的弱点不只在于密钥空间很小，还在于自然语言明文会泄露明显的统计特征。

## 总结

完成这题后，我对“暴力破解 + 统计评分”这种思路更熟悉了。单字节 XOR 从运算形式看很简单，但真正破解时要把几个步骤连起来：先把十六进制还原为 bytes，再枚举密钥，再对每个候选明文打分，最后在所有行里选最高分。过程中最容易出错的地方是只在某一行内部找最优密钥，却忘了还要和其他行的结果比较；另一个容易混淆的点是十六进制字符串本身不是密文 bytes，只是密文的显示形式。通过这题也能看出，密码分析里经常不需要直接知道原文内容，只要知道明文大概率属于哪种语言或格式，就可以用统计规律缩小搜索范围。

# 实验一：5.Implement repeating-key XOR

## 题目描述

本题要求实现重复密钥 XOR 加密。它和前面第 3 题的单字节 XOR 很像，区别在于密钥不再只有 1 个字节，而是一段较短的字符串。加密时不是把同一个字节一直用于整段明文，而是把密钥按顺序循环使用：明文第 1 个字节和密钥第 1 个字节异或，第 2 个字节和密钥第 2 个字节异或，第 3 个字节和密钥第 3 个字节异或；如果密钥用完，就从密钥开头重新开始。本题中给定的密钥是 `ICE`，需要把一段英文文本加密后输出十六进制密文。

## 过程

重复密钥 XOR 的核心仍然是异或运算。设明文字节序列为 `P`，密钥字节序列为 `K`，密钥长度为 `m`，那么第 `i` 个密文字节可以写成 `C[i] = P[i] xor K[i mod m]`。这里的 `i mod m` 用来让密钥循环，例如密钥 `ICE` 的 ASCII 字节分别是 `0x49`、`0x43`、`0x45`，明文第 0、3、6 个字节都会和 `0x49` 异或，第 1、4、7 个字节会和 `0x43` 异或，第 2、5、8 个字节会和 `0x45` 异或。由于 XOR 满足同一个值异或两次会抵消，所以这个函数既可以用于加密，也可以用于解密：密文再次和同一串重复密钥 XOR，就能恢复明文。

实现时我仍然把真正的运算放在 bytes 层完成。字符串只是人读起来方便，程序必须先把明文和密钥编码成字节，再逐字节异或。为了避免空密钥导致取模错误，我在函数开头检查了 `key` 是否为空。主要代码如下：

```python
def repeating_key_xor(data: bytes, key: bytes) -> bytes:
    if not key:
        raise ValueError("key must not be empty")

    return bytes(byte ^ key[index % len(key)] for index, byte in enumerate(data))
```

这段代码中，`enumerate(data)` 同时拿到当前字节的位置和数值，`index % len(key)` 负责计算当前应该使用密钥中的哪一个字节。外层的 `bytes(...)` 把每次异或得到的整数重新组装成字节序列。最后我又写了 `repeating_key_xor_hex`，把明文字符串和密钥字符串转成 bytes，调用核心函数后再用 `.hex()` 输出十六进制结果。运行 `repeating_key_xor.py` 后，程序输出为：

```text
0b3637272a2b2e63622c2e69692a23693a2a3c6324202d623d63343c2a26226324272765272a282b2f20430a652e2c652a3124333a653e2b2027630c692b20283165286326302e27282f
pass
```

这个结果与题目给出的密文一致，说明密钥循环的位置和字节异或的处理都是正确的。这里有一个细节是换行符也要参与加密，因为它也是明文中的一个字节，不能在处理字符串时随便删掉，否则后面的所有密钥对齐都会变化，最终密文也会不同。

## 总结

这道题让我把前几题的知识连起来了：第 1 题和第 2 题强调编码和 bytes 的区别，第 3 题说明单字节 XOR 可以通过穷举破解，第 5 题则把单字节密钥扩展成循环密钥。重复密钥 XOR 比单字节 XOR 的密钥空间更大，但它仍然不是现代意义上的安全加密方式，因为密钥会周期性重复。只要密文足够长，攻击者就可能从周期中推测密钥长度，再把问题拆成多个单字节 XOR 来分析。实现时我碰到的主要问题不是 XOR 本身，而是密钥下标的循环和明文换行符的保留；解决方法是完全按 bytes 的下标处理，不提前改变明文内容。

# 实验一：6.Break repeating-key XOR

## 题目描述

本题给出了一段 Base64 形式保存的密文，密文的真实字节并不是 Base64 本身，而是先经过重复密钥 XOR 加密后，再用 Base64 编码成可打印文本。我的理解是，这题要做的事情分成两层：第一层先把 Base64 还原成原始密文字节，第二层在不知道密钥的情况下破解重复密钥 XOR。重复密钥 XOR 的密钥会循环使用，所以它比单字节 XOR 更难一些；但只要能估计出密钥长度，就可以把密文按密钥位置拆开，每一组都会退化成前面做过的单字节 XOR 问题。

## 过程

重复密钥 XOR 的加密公式可以写成 `C[i] = P[i] xor K[i mod m]`，其中 `m` 是密钥长度。由于 `i mod m` 会周期性重复，同一列位置上的密文字节其实都被同一个密钥字节异或过。例如密钥长度为 3 时，第 0、3、6、9 个密文字节都使用密钥第 0 个字节，第 1、4、7、10 个密文字节都使用密钥第 1 个字节。这样一来，如果先猜出 `m`，就能把密文转置成 `m` 个小块，每个小块都可以用第 3 题的单字节 XOR 破解函数处理。

难点在于如何猜密钥长度。题目提示使用 Hamming distance，也就是两个等长字节串之间“不同 bit 的个数”。例如 `this is a test` 和 `wokka wokka!!!` 的 Hamming distance 应为 37。我在代码里先写了自检，确保这个基础函数没有问题：

```python
def hamming_distance(left: bytes, right: bytes) -> int:
    if len(left) != len(right):
        raise ValueError("inputs must have the same length")

    return sum((left_byte ^ right_byte).bit_count() for left_byte, right_byte in zip(left, right))
```

这里先对两个字节做 XOR，得到的结果中哪一位是 1，就说明原来两个字节在这一位不同。Python 的 `int.bit_count()` 可以直接统计整数二进制中 1 的个数，所以把所有位置的结果加起来，就是整段数据的 Hamming distance。为了比较不同密钥长度，距离还要除以 `keysize` 做归一化，否则长块天然会有更多不同 bit，不公平。

程序对 2 到 40 的可能密钥长度都计算归一化距离。为了让结果更稳定，我没有只比较前两个块，而是取前 8 个完整块，两两计算距离后求平均，距离越小代表这些块越像是由同一长度的重复密钥加密出来的。选出前几个候选长度后，再做转置分组：

```python
def transpose_blocks(ciphertext: bytes, keysize: int) -> list[bytes]:
    return [
        bytes(ciphertext[index] for index in range(position, len(ciphertext), keysize))
        for position in range(keysize)
    ]
```

这段代码的作用是把所有使用同一个密钥字节加密的位置收集到一起。比如 `position = 0` 时，取出的就是第 0、`keysize`、`2 * keysize` 等位置的密文字节。每一个转置后的块都相当于“固定单字节密钥 XOR”，所以可以直接调用前面第 3 题写好的 `break_single_byte_xor`，把每列最可能的单字节密钥拼起来，就是完整的重复密钥：

```python
key = bytes(
    break_single_byte_xor(block)[0]
    for block in transpose_blocks(ciphertext, keysize)
)
plaintext = repeating_key_xor(ciphertext, key)
score = score_english(plaintext) / len(plaintext)
```

最后程序会对候选密钥长度恢复出的全文再做一次英文评分，选出整体最像英文的结果。运行 `1.（6）break_repeating_key_xor.py` 后，Hamming distance 自检结果为 37，破解出的密钥长度为 29，密钥为 `Terminator X: Bring the noise`。用这个密钥对密文再次做重复密钥 XOR，就恢复出了正常英文歌词文本，开头是 `I'm back and I'm ringin' the bell`。这说明密钥长度估计、分组转置和单字节 XOR 破解三步都成功了。

## 总结

这题最重要的收获是看懂了“重复”为什么会带来弱点。重复密钥 XOR 表面上比单字节 XOR 强，因为密钥不再只有 256 种；但如果同一段短密钥在长文本里反复出现，密文也会保留周期性。Hamming distance 用来从统计上寻找这个周期，转置分组则把一个大问题拆成多个已经会解决的小问题。实现时我碰到的主要问题是密钥长度评分容易受偶然块影响，所以我改成多个块两两比较再平均；另一个细节是评分函数要对完整明文再筛一次，否则某个候选 keysize 的局部列评分可能看起来不错，但拼成全文后不一定最合理。整体来看，这道题把前面的 Base64、XOR、英文频率分析都串了起来，也让我更直观地理解了密码分析中“利用明文统计特征”和“利用密钥重复周期”的思路。

# 实验一：7.SHA1 哈希密码破解

## 题目描述

本题给出了管理员口令的 SHA1 哈希值，同时又给出了一张键盘痕迹图。我的理解是，单靠 SHA1 哈希本身很难直接反推出原口令，因为哈希函数的设计目标就是单向计算；但是键盘上的使用痕迹相当于泄露了一部分侧信道信息，它告诉我们口令大概由哪些按键组成。题目中的方向键是在登录成功后导航时留下的痕迹，不应该算入口令。真正有价值的是字母、数字、符号键以及 Shift 键，因为 Shift 会把同一个物理按键转换成另一个字符，例如德语键盘上 `Shift+8` 可以输入 `(`，`Shift+0` 可以输入 `=`，`Shift+q` 可以输入大写 `Q`。因此这题不是破解 SHA1 算法本身，而是利用键盘痕迹把候选口令空间缩小，再逐个计算 SHA1 进行比对。

## 过程

SHA1 会把任意长度的输入压缩成 160 bit 摘要，通常显示成 40 个十六进制字符。验证口令时，服务器并不需要保存明文口令，只需要保存 `hash = SHA1(password)`；用户登录时再对输入口令计算一次 SHA1，比较两个哈希是否相同。这个设计可以避免数据库中直接存放明文密码，但如果口令太短、没有加盐，攻击者拿到哈希后仍然可以离线尝试大量候选口令。题目给出的目标哈希为 `67ae1a64661ac8b4494666f58c4822408dd0a3e4`，键盘痕迹把候选字符限制到了少数几个按键，所以穷举成本很低。

我在代码中先建立“物理按键到可能输出字符”的映射。比如 `8` 这个键可能输出普通字符 `8`，也可能在按住 Shift 时输出 `(`；`q` 可能是 `q` 或 `Q`；`0` 可能是 `0` 或 `=`。然后程序枚举这些按键的所有排列，并对每个位置选择未按 Shift 或按 Shift 后的字符组合。每生成一个候选口令，就用 `hashlib.sha1(password.encode("utf-8")).hexdigest()` 计算摘要，并与题目给出的哈希值比较。主要实现如下：

```python
KEY_OUTPUT_OPTIONS = {
    "8": ("8", "("),
    "q": ("q", "Q"),
    "0": ("0", "="),
    "w": ("w", "W"),
    "i": ("i", "I"),
    "n": ("n", "N"),
    "*": ("*", "+"),
    "5": ("5", "%"),
}

def candidate_passwords():
    keys = tuple(KEY_OUTPUT_OPTIONS)
    for key_order in permutations(keys):
        option_groups = [KEY_OUTPUT_OPTIONS[key] for key in key_order]
        for chars in product(*option_groups):
            yield "".join(chars)
```

这里的 `permutations(keys)` 解决的是“这些按键先后顺序未知”的问题，`product(*option_groups)` 解决的是“同一个按键是否配合 Shift 使用未知”的问题。完整搜索空间大约是 `8! * 2^8`，也就是一千多万级别，实际程序很快就找到了匹配项。运行 `1.（7）sha1_keyboard_crack.py` 后输出结果为：

```text
target hash: 67ae1a64661ac8b4494666f58c4822408dd0a3e4
password: (Q=win*5
sha1(password): 67ae1a64661ac8b4494666f58c4822408dd0a3e4
attempts: 225
elapsed time: 0.000178 seconds
pass
```

因此本题的口令是 `(Q=win*5`。这个结果也和键盘痕迹能对应起来：`(` 来自 `Shift+8`，`Q` 来自 `Shift+q`，`=` 来自 `Shift+0`，后面的 `w`、`i`、`n`、`*`、`5` 都可以由图中的明显按键痕迹解释。程序不是把答案硬编码进去，而是在候选空间中通过 SHA1 校验找到唯一匹配的口令。

## 总结

这道题让我认识到，哈希函数的安全性和实际密码系统的安全性不是一回事。SHA1 虽然已经不适合新的安全协议，主要原因是碰撞攻击风险，但本题真正利用的不是 SHA1 的碰撞弱点，而是用户口令空间太小、哈希没有加盐、并且键盘痕迹泄露了候选字符。换句话说，攻击者不需要“反解 SHA1”，只要不断猜口令并计算哈希，猜到相同摘要就成功了。写代码时最关键的是把题目中的额外信息转化成搜索空间：排除方向键，把 Shift 带来的大小写和符号变化加入候选，再用哈希比对做最终判断。以后设计口令存储时，不能只保存裸 SHA1，更合理的做法是使用随机盐和专门的慢哈希算法，例如 bcrypt、scrypt 或 Argon2，这样即使哈希泄露，离线穷举的成本也会高很多。
