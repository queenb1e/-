# 实验二：1.MTC3 AES key -- encoded in the machine readable zone of a European ePassport

## 题目描述

本题要求根据电子护照机读区 MRZ 中的信息恢复 AES 密钥，并解开题目给出的 Base64 密文。我的理解是，MRZ 不是直接把 AES 密钥写出来，而是提供了证件号、出生日期、有效期以及校验位等字段。程序需要先补全缺失的 MRZ 校验位，再按照题目给出的派生规则计算 `K_seed` 和 `K_enc`，最后用 AES-CBC 解密密文。

## 过程

MRZ 校验位使用权重 `7, 3, 1` 循环计算。数字按本身取值，字母 `A-Z` 映射为 `10-35`，填充符 `<` 记为 0。题目中缺失的是有效期字段后面的校验位，因此我先对 `111116` 计算校验位，得到 `7`，恢复后的 MRZ 为：

```text
12345678<8<<<1110182<1111167<<<<<<<<<<<<<<<4
```

随后提取 `证件号+校验位`、`出生日期+校验位`、`有效期+校验位` 作为 MRZ information，计算 `SHA1(MRZ_information)` 的前 16 字节作为 `K_seed`。再对 `K_seed || 00000001` 计算 SHA1，取前 16 字节并调整奇校验位，得到 AES 密钥 `K_enc`。核心代码如下：

```python
def derive_k_seed(mrz: str) -> bytes:
    mrz_information = (mrz[:10] + mrz[13:20] + mrz[21:28]).encode("ascii")
    return hashlib.sha1(mrz_information).digest()[:16]


def derive_kenc(k_seed: bytes) -> bytes:
    digest = hashlib.sha1(k_seed + bytes.fromhex("00000001")).digest()
    return set_odd_parity(digest[:16])
```

运行 `2.（1）mtc3_aes_mrz.py` 后，程序输出：

```text
K_seed: a095f0fdfe51e6ab3bf5c777302c473e
K_enc: ea8645d97ff725a898942aa280c43179
plaintext: Herzlichen Glueckwunsch. Sie haben die Nuss geknackt. Das Codewort lautet: Kryptographie!
```

## 总结

这题让我认识到，密钥派生的安全性不仅取决于 AES 本身，还取决于输入信息是否足够随机。MRZ 中的证件号、日期和校验位格式固定，熵并不高，所以电子护照协议必须配合访问控制和更完整的安全机制。实现时最容易出错的是字段下标和校验位位置，解决方法是先把 MRZ 字符串分段打印验证，再进行哈希派生。

# 实验二：2.Implement PKCS#7 padding

## 题目描述

本题要求实现 PKCS#7 填充。给定任意长度的明文和块大小，程序要在末尾补若干字节，使总长度刚好变成块大小的整数倍。补上的每个字节值都等于补字节的数量。

## 过程

如果块大小为 20，明文 `YELLOW SUBMARINE` 长度为 16，则还差 4 字节，所以填充结果应为 `YELLOW SUBMARINE\x04\x04\x04\x04`。如果明文本来已经是块大小的整数倍，也必须额外补一个完整块，避免解密时无法判断最后一个字节到底是数据还是填充。

核心代码如下：

```python
def pkcs7_pad(data: bytes, block_size: int = 16) -> bytes:
    padding_length = block_size - (len(data) % block_size)
    if padding_length == 0:
        padding_length = block_size
    return data + bytes([padding_length]) * padding_length
```

运行 `2.（2）pkcs7_padding.py` 后输出：

```text
b'YELLOW SUBMARINE\x04\x04\x04\x04'
pass
```

## 总结

PKCS#7 的规则看起来很简单，但它是 CBC 等分组模式能处理非整块明文的基础。实现时要注意“刚好整块也要补一整块”这个边界条件，否则解密端会出现歧义。

# 实验二：3.Implement CBC mode

## 题目描述

本题要求在已有 AES-ECB 单块加解密的基础上实现 CBC 模式。CBC 的核心是“链式反馈”：每个明文块在 AES 加密前都要先和前一个密文块异或，第一个块则和 IV 异或。

## 过程

CBC 加密公式为 `C_i = AES_K(P_i xor C_{i-1})`，其中 `C_0` 使用 IV。解密公式为 `P_i = AES_K^{-1}(C_i) xor C_{i-1}`。因此 CBC 本身并不是新的分组密码，而是把已有分组密码按块连接起来的一种工作模式。

我在 `crypto_utils.py` 中实现了纯 Python AES-128，并在此基础上实现 CBC：

```python
def aes_cbc_encrypt(data: bytes, key: bytes, iv: bytes, pad: bool = True) -> bytes:
    plaintext = pkcs7_pad(data, 16) if pad else data
    previous = iv
    result = []
    for block in split_blocks(plaintext):
        encrypted = aes_encrypt_block(xor_bytes(block, previous), key)
        result.append(encrypted)
        previous = encrypted
    return b"".join(result)
```

运行 `2.（3）implement_cbc_mode.py` 后，解密得到的文本与原文完全一致：

```text
recovered: CBC mode chains every plaintext block with the previous ciphertext block. This test verifies encryption and decryption with a zero IV.
pass
```

## 总结

CBC 模式比 ECB 更能隐藏重复明文块，因为同一个明文块在不同位置会受前一个密文块影响，通常不会产生相同密文块。但 CBC 对 IV 和填充处理很敏感，如果 IV 可预测或填充验证暴露错误信息，就可能引出后续的攻击。

# 实验二：4.An ECB/CBC detection oracle

## 题目描述

本题要求编写一个加密预言机：它随机生成密钥，在用户输入前后添加 5 到 10 字节随机数据，然后随机选择 ECB 或 CBC 加密。攻击者只能看到密文，需要判断使用的是 ECB 还是 CBC。

## 过程

ECB 的明显弱点是相同明文块会产生相同密文块。因此我向预言机输入大量相同字节，例如 64 个 `A`。即使前后有少量随机字节，中间仍然大概率出现多个完全相同的 16 字节明文块。如果密文中存在重复块，就判断为 ECB；否则判断为 CBC。

核心判断代码如下：

```python
def guess_mode(ciphertext: bytes) -> str:
    return "ECB" if detect_ecb(ciphertext) else "CBC"
```

运行 `2.（4）ecb_cbc_detection_oracle.py` 后，20 次随机试验全部判断正确：

```text
correct guesses: 20/20
pass
```

## 总结

这题说明 ECB 的问题不是 AES 算法本身弱，而是工作模式泄露了块级重复结构。只要攻击者能构造重复明文，ECB 就很容易被识别出来。这也是实际系统中不应使用 ECB 加密结构化数据的原因。

# 实验二：5.Byte-at-a-time ECB decryption (Simple)

## 题目描述

本题给出一个 ECB 加密预言机：攻击者可以输入任意数据，预言机会把输入拼接在一个未知字符串前面，再用固定随机密钥做 AES-ECB 加密。目标是在不知道密钥的情况下恢复未知字符串。

## 过程

攻击思路利用 ECB 的确定性。先通过密文长度变化检测块大小为 16，再输入重复块确认模式为 ECB。随后每次构造 `15 - 已知长度 mod 16` 个 `A`，让未知字符串的下一个字节落在当前块末尾。攻击者枚举 0 到 255 的候选字节，构造字典，把“已知前缀 + 已知明文 + 候选字节”的密文块映射回候选值。真实预言机输出的目标块能在字典中命中，因此可以逐字节恢复明文。

核心代码如下：

```python
for _ in range(unknown_length):
    pad_length = block_size - 1 - (len(discovered) % block_size)
    prefix = b"A" * pad_length
    block_index = len(discovered) // block_size
    target_block = split_blocks(oracle(prefix), block_size)[block_index]

    dictionary = {}
    known_prefix = prefix + discovered
    for candidate in range(256):
        block = split_blocks(oracle(known_prefix + bytes([candidate])), block_size)[block_index]
        dictionary[block] = candidate

    discovered += bytes([dictionary[target_block]])
```

运行 `2.（5）byte_at_a_time_ecb_simple.py` 后恢复出：

```text
Rollin' in my 5.0
With my rag-top down so my hair can blow
The girlies on standby waving just to say hi
Did you stop? No, I just drove by

pass
```

## 总结

这题非常直观地展示了 ECB 在“可选择明文”场景下的危险性。攻击者不需要知道密钥，只要能反复调用加密接口，就可以利用 ECB 的固定块映射一点点建立字典，恢复接口内部拼接的秘密数据。

Cryptopals Crypto Challenges, Set 2 Challenge 12: https://cryptopals.com/sets/2/challenges/12

Wikipedia, Block cipher mode of operation: https://en.wikipedia.org/wiki/Block_cipher_mode_of_operation



# 实验二：6.ECB cut-and-paste

## 题目描述

本题要求模拟一个用户资料加密接口。接口会把邮箱转换成 `email=...&uid=10&role=user` 的格式，再使用 AES-ECB 加密。目标是在不能直接输入 `&` 和 `=` 的情况下，构造密文使解密后的角色变成 `admin`。

## 过程

ECB 的每个块独立加密，因此可以把不同密文中的块剪切拼接。我的做法是先构造一个邮箱，使 `admin` 加上合法 PKCS#7 填充正好占满一个完整块；再构造另一个普通资料，使 `role=` 后面刚好从新块开始。最后把普通资料前两个块和 `admin` 块拼接起来。

核心代码如下：

```python
def forge_admin_profile() -> bytes:
    admin_block_email = "A" * 10 + "admin" + chr(11) * 11
    admin_block = encrypt_profile(admin_block_email)[16:32]

    normal_profile = encrypt_profile("B" * 13)
    return normal_profile[:32] + admin_block
```

运行 `2.（6）ecb_cut_and_paste.py` 后输出：

```text
{'email': 'BBBBBBBBBBBBB', 'uid': '10', 'role': 'admin'}
pass
```

## 总结

这题说明加密并不等于完整性保护。即使攻击者不知道 AES 密钥，只要 ECB 块可以被剪切拼接，结构化明文就可能被重组出新的语义。实际系统中应使用带认证的加密模式，或者至少对密文加 MAC。

# 实验二：7.Byte-at-a-time ECB decryption (Harder)

## 题目描述

本题是在第 5 题基础上的增强版。预言机会在用户输入前额外拼接一段固定但长度未知的随机前缀，再拼接未知字符串并用 ECB 加密。目标仍然是在不知道密钥和随机前缀的情况下恢复未知字符串。

## 过程

随机前缀会破坏我们对块边界的控制，因此第一步是对齐边界。我依次尝试不同长度的 `A`，后面接两个完整块的 `B`。当密文中出现两个相邻重复块时，说明这两个 `B` 块已经被对齐到完整 AES 块边界。此时可以得到需要补多少个 `A`，以及用户可控内容从第几个密文块开始。

核心对齐代码如下：

```python
def find_prefix_alignment(block_size: int = 16) -> tuple[int, int]:
    for pad_length in range(block_size):
        probe = b"A" * pad_length + b"B" * (block_size * 2)
        blocks = split_blocks(oracle(probe), block_size)
        for index in range(len(blocks) - 1):
            if blocks[index] == blocks[index + 1]:
                return pad_length, index
    raise ValueError("could not align controlled input")
```

完成对齐后，第 5 题的逐字节字典攻击可以复用，只是目标块下标要加上随机前缀占用的块数。运行 `2.（7）byte_at_a_time_ecb_harder.py` 后，程序成功恢复出同一段未知字符串：

```text
Rollin' in my 5.0
With my rag-top down so my hair can blow
The girlies on standby waving just to say hi
Did you stop? No, I just drove by

pass
```

## 总结

这题的关键不是重新发明攻击，而是先恢复块边界控制权。随机前缀只要是固定的，就可以通过重复块探测对齐；一旦对齐成功，ECB 的确定性弱点仍然可以被利用。

# 实验二：8.PKCS#7 padding validation

## 题目描述

本题要求实现 PKCS#7 填充校验。程序要能识别合法填充并去掉填充，同时对非法填充抛出错误。

## 过程

PKCS#7 校验不能只看最后一个字节的数值，还必须检查最后 `n` 个字节是否全都等于 `n`。例如 `\x04\x04\x04\x04` 是合法的 4 字节填充，但 `\x01\x02\x03\x04` 虽然最后一个字节也是 4，却不是合法填充。

核心代码如下：

```python
def pkcs7_unpad(data: bytes, block_size: int = 16) -> bytes:
    padding_length = data[-1]
    if padding_length == 0 or padding_length > block_size:
        raise ValueError("invalid PKCS#7 padding")
    if data[-padding_length:] != bytes([padding_length]) * padding_length:
        raise ValueError("invalid PKCS#7 padding")
    return data[:-padding_length]
```

运行 `2.（8）pkcs7_padding_validation.py` 后，三组样例结果都正确：

```text
b'ICE ICE BABY\x04\x04\x04\x04' -> b'ICE ICE BABY'
pass
b'ICE ICE BABY\x05\x05\x05\x05' -> invalid PKCS#7 padding
pass
b'ICE ICE BABY\x01\x02\x03\x04' -> invalid PKCS#7 padding
pass
```

## 总结

这题补上了第 2 题的逆操作，也为理解 padding oracle 攻击做准备。填充验证应该严格、统一，并避免把错误差异暴露给攻击者，否则会变成攻击侧信道。

# 实验二：9.CBC bit flipping attacks

## 题目描述

本题要求利用 CBC 的可篡改性，在不能直接输入 `;` 和 `=` 的情况下，使解密后的明文中出现 `;admin=true;`。加密接口会过滤用户输入中的分隔符，但不提供密文认证。

## 过程

CBC 解密时有 `P_i = AES^{-1}(C_i) xor C_{i-1}`。因此如果攻击者修改前一个密文块 `C_{i-1}` 的某个 bit，就会按同样位置翻转当前明文块 `P_i` 的 bit。虽然这会破坏前一个明文块，但只要目标是让当前块出现特定字符串，就可以接受这种局部破坏。

我先输入一整块 `A`，再修改它前一个密文块的对应字节，使解密后从 `A` 变成 `;admin=true;AAAA`：

```python
for index, desired_byte in enumerate(desired):
    ciphertext[previous_block_offset + index] ^= ord("A") ^ desired_byte
```

运行 `2.（9）cbc_bit_flipping_attacks.py` 后输出：

```text
admin: True
pass
```

## 总结

这题说明 CBC 加密只能提供保密性，不能自动提供完整性。攻击者即使不知道密钥，也可能通过翻转密文 bit 精确控制下一块明文的变化。实际系统中应使用认证加密，例如 AES-GCM，或者采用 Encrypt-then-MAC。
