# 实验四：1.RSA 加密体制破译

## 题目描述

本题来自首届（2016）全国高校密码数学挑战赛赛题三，要求根据截获的 RSA 加密帧恢复通关密语和尽可能多的 RSA 参数。每个帧文件由三段 1024 bit 十六进制数据组成，格式为 `N | e | c`，其中 `N` 是 RSA 模数，`e` 是公开指数，`c = m^e mod N` 是密文。

题目中的明文不是直接作为普通字符串加密，而是先被填充成 512 bit：高位 64 bit 是通信标志位，中间有 32 bit 通信序号，然后补 0，最后 64 bit 保存最多 8 个明文字符。我的理解是，这道题不是要暴力分解所有 1024 bit RSA 模数，而是要利用参数生成和使用过程中的错误，例如共用模数、共享素因子、弱随机数生成器、过光滑素因子和重复发送。

## 过程

我先编写 `parse_frame` 按固定长度拆分每个 Frame 文件：

```python
def parse_frame(path: Path) -> Frame:
    data = path.read_text(encoding="ascii").strip()
    return Frame(
        index=int(path.name.removeprefix("Frame")),
        name=path.name,
        n=int(data[:256], 16),
        e=int(data[256:512], 16),
        c=int(data[512:], 16),
    )
```

拆帧后，我先对所有模数两两计算 `gcd(N_i, N_j)`。如果两个模数共享同一个素因子，那么 `gcd` 会直接给出该素因子，从而可以计算 `phi(N)` 和私钥 `d`。本题中 `Frame1` 与 `Frame18` 共享了一个 512 bit 素因子，因此可以直接解密两帧。

第二类漏洞是共模攻击。`Frame0` 和 `Frame4` 使用了相同的模数 `N`，并且两个公开指数互素。如果它们加密的是同一明文，则存在整数 `a, b` 使 `a*e1 + b*e2 = 1`，因此：

```text
m = c1^a * c2^b mod N
```

当 `a` 或 `b` 为负数时，先对对应密文求模逆。代码中使用扩展欧几里得算法求出系数，并成功恢复出序号 0 的明文块 `My secre`。

第三类漏洞来自素数生成器。题目说明素数 `p` 由随机数发生器生成，附件中的参数表现出明显的 16 bit 线性同余序列特征。我使用如下递推枚举 16 bit 种子，并将生成出的候选大整数与 `N` 求 gcd：

```python
x = seed
value = x
for _ in range(31):
    x = (365 * x - 1) & 0xFFFF
    value = (value << 16) | x
factor = gcd(value, n)
```

如果 `factor` 满足 `1 < factor < n`，说明该候选数就是 `N` 的一个素因子。大部分帧可以在 512 bit 候选长度处直接找到因子；`Frame13` 和 `Frame17` 的候选因子需要在 520 bit 附近逐 bit 右移裁剪后命中。

第四类漏洞是 Pollard p-1 攻击。若某个素因子 `p` 的 `p-1` 比较光滑，即分解后主要由较小素因子组成，则可以通过不断计算 `a = a^j mod N` 并检查 `gcd(a-1, N)` 找到因子：

```python
def pollard_p_minus_1(n: int, bound: int = 500_000) -> int | None:
    value = 2
    for exponent in range(2, bound + 1):
        value = pow(value, exponent, n)
        if exponent % 1000 == 0:
            factor = gcd(value - 1, n)
            if 1 < factor < n:
                return factor
    return None
```

```text

```

## 总结

这次实验让我更直观地理解了 RSA 的安全性不仅取决于模数长度，还取决于参数是否正确生成和正确使用。1024 bit 模数在一般情况下不容易直接分解，但如果多个模数共享素因子、同一模数重复使用、素数由可预测的随机数发生器产生，或者 `p-1` 太光滑，那么攻击者仍然可以恢复私钥和明文。

实现过程中最容易出错的是明文拼接顺序。Frame 文件名表示接收序号，不等于通信序号；真正的分片顺序藏在 512 bit 填充中的 32 bit 字段里。解决方法是解密后解析填充结构，按通信序号排序，并对重复发送的分片去重。
